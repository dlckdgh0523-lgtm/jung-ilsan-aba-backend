## 4. 리소스별 실제 계약 (프론트가 읽고/쓰는 필드)

### brand / site (Chrome.jsx, Sections.jsx)
- 읽음: `nameKo, tagline, address, phone, fax, hours, kakaoId, coords{lat,lng}, kakaoMapUrl, naverMapUrl` (옵션 `kakaoMapAppKey`). `kakaoQrUrl`은 코드상 안 읽음(QR은 정적 asset 경로 하드코딩).
- nav 항목: `{id, label, visible}` (프론트엔 `order` 없음; 관리자 메뉴뷰는 배열 순서로 정렬). 기본 nav: about/programs/therapists/notices/gallery/contact.
- sections(섹션 ON/OFF) 관리자 토글 id: hero, about, programs, therapists, notices, gallery, **homemap**(홈 지도; 스펙엔 없음), contact.

### hero (HeroSlider.jsx, Admin Hero)
- 읽음: `id, image, eyebrow, title(\n 포함), subtitle, buttonText, buttonLink`, 옵션 `buttonText2, buttonLink2`(슬라이드 2에 존재).
- 관리자 작성: `image(photo), eyebrow, title, subtitle, buttonText, buttonLink` (버튼2는 폼에 없음).
- ⚠ API_SPEC은 `imageUrl`. **프론트는 `image`.**

### about (Sections AboutSection, Admin About)
- 읽음: `eyebrow, title(\n), body[]`(문단 배열). `lead[]`/`values[]`는 data엔 있으나 현재 UI 미렌더(보존).
- 관리자 작성: about의 `title, body[]` + brand의 `nameKo,tagline,address,phone,kakaoId,hours`를 **한 폼에서 합쳐서** 저장(=/site + /about 동시 갱신 필요).
- ⚠ API_SPEC values: `[{icon,title,desc}]`. data.js values: `[{ko,en,desc}]`. **data.js 형태 채택.**

### director (Sections DirectorProfile, Admin Director)
- 객체명: data.js `directorProfile` (스펙은 `/director`).
- 읽음(공개): `sub, photo, certifications[{code,desc}], education[], societies[], awards[], career[{period,text}], papers[{year,title}], training[]`. (AboutSection 배경에 `photo` 사용)
- 관리자 작성: `photo, name, role, sub, certifications[{code,desc}], education[], **organizations**[], career[{period,text}], awards[], training[]`.
- ⚠⚠ **프론트 내부 불일치**: 공개 화면은 `societies`를 읽고, 관리자 폼은 `organizations`로 씀. + 공개는 `papers`를 읽지만 관리자 폼엔 papers 없음.
- ⚠ API_SPEC: `photoUrl`, `organizations`. data.js: `photo`, `societies`(+papers).

### programs (Sections ProgramsSection/ProgramDetail, Admin Programs)
- 읽음/작성: `id, photo, title, ageRange, desc, icon, tone(orange|green|yellow|green-soft), tags[], meta, detail{intro, sections[{heading,body}]}`.
- ⚠ API_SPEC: `photoUrl`. data.js: `photo`. (tone enum 동일, detail 동일)

### therapists (Sections, Admin Therapists)
- 읽음(공개): `id, photo, name, role, summary, education[], career[{period,text}], certifications[], teaching[], completion`.
- 관리자 작성: `photo, name, role, summary, education[], career[], certifications[], completion` (teaching 폼 없음; tone은 data엔 "blue"지만 UI/폼 미사용).
- ⚠ API_SPEC: `photoUrl`(+tone enum blue 없음). data.js: `photo` + `teaching[]` 추가.

### notices (Sections, Admin Notices)
- 읽음/작성: `id, title, body, date(문자열 "2026.05.18" 점 구분), pinned(bool), views(number)`.
- 관리자 신규 기본 date = `YYYY.MM.DD`(오늘), views=0.
- ⚠ API_SPEC date는 "2026-05-12"(하이픈). **프론트는 점 표기, 그대로 표시** → date를 자유 문자열로 저장(서버가 형식 강제 X) 권장.

### gallery (Sections GallerySection, Admin Gallery)
- 읽음/작성: `id, src(이미지), title(캡션), span(1|2)`.
- ⚠ API_SPEC: `imageUrl, caption, tags[]`. data.js: `src, title, span`(tags 없음).

### popups (Sections PopupHost/PopupModal, Admin Popups)
- 읽음/작성: `id, title, content, imageUrl, linkUrl, startAt, endAt, isActive, allowHideToday, createdAt, updatedAt, deletedAt`.
- ⚠⚠ **날짜 형식**: 프론트가 `new Date(p.startAt + "T00:00:00")`, `new Date(p.endAt + "T23:59:59")` 로 처리 → **startAt/endAt는 반드시 "YYYY-MM-DD"(date-only) 문자열**. API_SPEC의 풀 ISO datetime을 그대로 주면 깨짐. 관리자 폼도 `<input type=date>`(YYYY-MM-DD).
- 상태계산(관리자 popupStatus): isActive=false→비활성, now<start→예약중, now>end→종료됨, else 노출중. (스펙: scheduled/active/ended/deleted)
- 공개 노출조건(popupIsShowable): `!deletedAt && isActive && start<=now<=end && !오늘숨김`.

### consultations (Sections ContactSection submit, Admin ConsultationsView)
- 프론트 현재 payload(localStorage): `{id, ts(ms), parent, phone, age(문자열 "만 4세"), topic(프로그램 제목 텍스트), note, read(bool)}`.
- 관리자 표시 컬럼: 부모님(parent), 연락처(phone), 연령(age), 분야(topic), 접수시각(ts), 상태(read), 메모(note).
- ⚠ API_SPEC consultations는 더 리치: `parent, child{age,diagnosis}, phone, email, topic, preferred[], message, privacyConsent, status(new|contacted|done), isRead, createdAt, ip`.
- 결정: 스펙 스키마를 기준으로 저장하되, **프론트 호환 alias**(`note`↔`message`, `read`↔`isRead`, `age`(평면 문자열)·`ts`(epoch ms) 동시 제공) 응답 포함. 컨택폼/관리자 fetch 교체는 마이그레이션 노트로 안내.

### centerInfo (data.js + Sections CenterInfoBlock) — 스펙에 없음
- 형태: `{eyebrow, title, sub, highlights[{icon,title,desc}], earlyClasses[{name,time,consult}], individualSessions[{name,time,consult}], closingNote}`.
- 결정: `GET/PUT /center-info` 엔드포인트 추가(공개 GET / 관리자 PUT). 부트스트랩에도 합류.

### privacy (Sections PrivacyPolicyPage/PrivacyConsentPage)
- 현재는 페이지에 텍스트 하드코딩. API_SPEC §14: `/privacy/policy`, `/privacy/consent` → `{version, body}`(마크다운). 그대로 구현.

### auth / stats / realtime
- 로그인 데모 계정: `admin` / `aba1234` (시드 관리자 기본값으로 사용, env로 override).
- stats: 관리자 대시보드는 현재 난수 시뮬. API_SPEC §13 구현(POST /stats/hit, GET /stats/summary, /stats/range, concurrent=최근90초 고유 session).
- realtime: 상담 신규 push(SSE `/realtime/consultations?token=`), ping heartbeat, `/realtime/stats`(30초). 프론트는 현재 BroadcastChannel; 마이그레이션 시 EventSource로 교체.

## 5. ✅ Reconciliation 규칙 (확정됨 — 2026-06-02 사용자 결정)
**사용자 결정: "API_SPEC.md(필드명)는 신경쓰지 말고 실제 프론트와 비교해서 작업." → 프론트(data.js/*.jsx)가 SSOT.**
1. **응답 필드명 = 프론트가 실제 읽는 이름 그대로** (image, photo, src, title, span, societies, papers, teaching, centerInfo, date-only popups...). API_SPEC의 imageUrl/photoUrl/caption/organizations 등 명칭은 따르지 않음.
2. **관리자 운영에 필요한 추가 필드는 보존**: `visible`(RowActions 토글이 읽음), `order`(정렬), `createdAt/updatedAt`(timestamps), soft delete `deletedAt`. 프론트 표시 코드는 이를 무시하므로 안전.
3. **프론트에만 있는 필드/개념은 전부 구현**: centerInfo(+/center-info 엔드포인트), hero `buttonText2/buttonLink2`, gallery `span`, director `papers`, therapist `teaching`, sections `homemap`.
4. **director societies/organizations 불일치 처리**: 단일 컬럼에 저장하고 **응답에 `societies`와 `organizations` 둘 다 동일 배열로 노출**(공개 화면=societies, 관리자 폼=organizations 둘 다 무수정 동작). 입력은 둘 중 무엇이 와도 수용.
5. **consultations**: 프론트 컨택폼이 보내는/관리자 테이블이 읽는 필드 그대로 우선 — 응답에 `parent, phone, age(문자열), topic, note, read, ts(epoch ms)` 포함. 내부엔 status/isRead/createdAt/ip 등 운영 필드도 저장하고 별칭으로 함께 제공.
6. 경로/페이지네이션/공통 규약/soft-delete·order·visibility 패턴 등 **구조는 API_SPEC을 참고**(유용하므로) 하되, 필드명 충돌 시 프론트 승.

## 7. 보안 요구 (API_SPEC §18 + 프롬프트 §9)
Helmet, CORS 화이트리스트, ValidationPipe(whitelist+transform), 관리자 라우트 JwtAuthGuard+RolesGuard, `POST /consultations` IP당 분당 5회 rate limit, privacyConsent 미체크 422, 업로드 mimetype/확장자 화이트리스트+5MB+1600px 리사이즈, soft delete 우선/hard delete confirm, SSE 토큰만료 401+close, env 분리(.env.example).

## 8. 응답 규약
- 목록 envelope: `{items, page, pageSize, total}`. 쿼리: `page,pageSize(≤100),sort(field:asc|desc),q,visible(true|false|all),trashed(only|with)`.
- 단건/객체는 리소스 그대로. 순서변경 `PUT /<r>/order {ids[]}`→204. 가시성 `PATCH /<r>/{id}/visibility {visible}`→200. 복구 `POST /<r>/{id}/restore`. hard delete `DELETE ?hard=true`.
- 오류: `{ "error": { "code, message, fields? } }`, HTTP 400/401/403/404/409/422/500. 전역 ExceptionFilter + 전역 ResponseInterceptor.

## 9. 아키텍처 / 모듈 계획
`src/`: auth, site(+nav, sections, center-info), hero, about, director, programs, therapists, notices, gallery, popups, consultations, realtime, stats, privacy, uploads, common(interceptors/filters/guards/decorators/dto/pagination/base-crud), prisma, config. 단일 객체 리소스(site/about/director/center-info/privacy)는 singleton 행. 표준 CRUD 리소스는 BaseCrudService 재사용(order/visible/soft-delete/restore/pagination).
